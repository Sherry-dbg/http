GET 获取资源 — 最常用的HTTP方法

）：不会被浏览器显示 下面这段注释解释 GET 方法的特点 --> 

// HTTP GET 请求格式
GET /api/posts/1 HTTP/1.1
Host: jsonplaceholder.typicode.com
Accept: application/json
User-Agent: Mozilla/5.0
（请求体为空 —— GET 没有 body）

POST 提交数据 — 创建新资源 


// HTTP POST 请求格式
POST /api/posts HTTP/1.1
Host: jsonplaceholder.typicode.com
Content-Type: application/json; charset=UTF-8
Content-Length: 62

// 请求体 (JSON 格式的数据)
{
  "title": "Hello HTTP",
  "body": "这是一条新帖子",
  "userId": 1
}

PUT 全量更新 — 替换整个资源 


// HTTP PUT 请求格式
PUT /api/posts/1 HTTP/1.1
Host: jsonplaceholder.typicode.com
Content-Type: application/json; charset=UTF-8

{
  "id": 1,
  "title": "更新后的标题",
  "body": "完整替换内容",
  "userId": 1
}

DELETE 删除资源 


// HTTP DELETE 请求格式
DELETE /api/posts/1 HTTP/1.1
Host: jsonplaceholder.typicode.com
（请求体通常为空 —— DELETE 不需要 body）

PATCH 部分更新 — 只修改资源的一部分字段 


// HTTP PATCH 请求格式（只发送需要修改的字段）
PATCH /api/posts/1 HTTP/1.1
Host: jsonplaceholder.typicode.com
Content-Type: application/json; charset=UTF-8

{
  "title": "只更新标题"
  // 注意：这里只发了 title，body 和 userId 都不发
  // 服务器只更新 title 字段，其他字段保持不变
}

HEAD 获取响应头 — 不返回响应体 


// HTTP HEAD 请求格式
HEAD /api/posts/1 HTTP/1.1
Host: jsonplaceholder.typicode.com
（格式和 GET 一样，但服务器不返回 body）

// 服务器响应（只有头部，body 为空）：
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 245
Cache-Control: max-age=43200
（注意：没有 body 内容，只有元数据）

OPTIONS 预检请求 — 查询服务器支持的方法 


// HTTP OPTIONS 请求格式（CORS 预检请求）
OPTIONS /api/posts HTTP/1.1
Host: jsonplaceholder.typicode.com
Origin: https://example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type

// 服务器响应（列出所有允许的方法）：
HTTP/1.1 204 No Content
Allow: GET, POST, PUT, DELETE, PATCH, OPTIONS
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type

常用 HTTP 状态码一览表

状态码原因短语含义说明
200 OK 请求成功！GET 返回资源数据，POST 返回处理结果
201 Created 资源已创建（POST 成功后返回），响应头 Location 指向新资源的 URL
204 No Content 请求成功但没有返回内容（DELETE 操作常用），响应体为空
301 Moved Permanently 资源"永久"搬家到新 URL，浏览器会记住新地址，搜索引擎更新索引
302 Found 资源"临时"搬家，浏览器不记新地址，下次还用旧地址
304 Not Modified 资源没变，请用本地缓存（省流量），配合 If-None-Match / ETag 头使用
400 Bad Request 你的请求有语法错误或参数不对，服务器看不懂
401 Unauthorized 需要登录！你没带 Token 或 Token 过期了
403 Forbidden 服务器拒绝：你登录了，但没权限（如普通用户访问管理员页面）
404 Not Found 你要的资源不存在（最常见的错误，URL 打错了最常见）
405 Method Not Allowed 方法不对！如对只读资源用了 POST（应该用 GET）
429 Too Many Requests 你请求太快了，触发限流！稍等再试
500 Internal Server Error 服务器代码崩了（bug、配置错了等），不是你的问题
502 Bad Gateway 网关/代理收到上游服务器的无效响应
503 Service Unavailable 服务器暂时不可用（过载、在维护），等会再试

常用 HTTP 请求头 (Request Headers)

请求头示例值说明
Accept text/html, application/json 告诉服务器：我能接收这些格式的响应（MIME 类型）
Authorization Bearer eyJhbGciOi... 携带认证令牌（JWT Token），证明"我是谁"
Content-Type application/json; charset=utf-8 告诉服务器：我发的请求体是什么格式（POST/PUT 时才需要）
Cookie sessionId=abc123; theme=dark 把之前服务器设置的 Cookie 发回去（登录状态等）
User-Agent Mozilla/5.0 (Windows NT 10.0) ... 告诉服务器：我用的是什么浏览器和操作系统
Referer https://www.google.com/ 当前请求是从哪个页面跳过来的（注意拼写是错的，历史遗留）
Cache-Control no-cache 控制缓存行为（no-cache 表示要验证后才能用缓存）
If-None-Match "abc123" 配合 ETag 做条件请求：版本相同返回 304，不同返回 200+新内容
Origin https://mysite.com 跨域请求时告知服务器：我的来源网站是 xxx
Host api.example.com 目标服务器域名，HTTP/1.1 必须有的头（一台服务器托管多网站时分辩用哪个）

常用 HTTP 响应头 (Response Headers)

响应头示例值说明
Content-Type text/html; charset=utf-8 告诉浏览器：响应内容是什么类型（HTML/JSON/图片等）和编码
Content-Length 348 响应体的字节数（文件大小）
Set-Cookie sessionId=abc; HttpOnly; Secure 要求浏览器保存 Cookie（HttpOnly=JS 不可读，Secure=仅 HTTPS 传输）
Cache-Control max-age=3600, public 告诉浏览器：这个内容可以缓存 3600 秒（1小时）
ETag "33a64df5514" 资源的"指纹"（通常是哈希值），内容变了指纹就变，用于缓存校验
Location /posts/42 重定向目标地址（201 创建成功指向新资源，301/302 指向跳转地址）
Access-Control-Allow-Origin * 或 https://a.com CORS：允许哪些网站的 JS 跨域访问我（* = 任意网站）
Last-Modified Wed, 21 Oct 2025 07:28:00 GMT 资源最后一次修改的时间
X-Frame-Options DENY 安全：禁止本页面被嵌入到 iframe 中（防止点击劫持攻击）

图片通过 HTTP 加载的完整演示


// HTML 中的 img 标签会自动发起 HTTP GET 请求
<img src="https://picsum.photos/400/300" alt="示例图片">

// 浏览器实际发送的 HTTP 请求（自动生成）：
GET /400/300 HTTP/1.1
Host: picsum.photos
Accept: image/avif,image/webp,image/apng,image/*,*/*
// 注意：浏览器自动添加了 Accept 头，表示它能接收这些图片格式

// 服务器响应：
HTTP/1.1 200 OK
Content-Type: image/jpeg          ← 告诉浏览器这是 JPEG 图片
Content-Length: 48231              ← 图片文件大小：约 48KB
Cache-Control: max-age=86400      ← 可以缓存 24 小时（86400秒）
（响应体：图片的二进制数据，浏览器解析后显示）

一次完整的 HTTP 请求-响应交换


/* ====== 第1步：客户端 → 服务器 (HTTP 请求报文) ====== */

POST /api/users HTTP/1.1   ← ① 请求行：POST方法 + /api/users路径 + HTTP/1.1协议
Host: api.example.com         ← ② 请求头开始：目标主机域名
Content-Type: application/json ← 告诉服务器：请求体是 JSON 格式
Content-Length: 57            ← 请求体的字节数（用于服务器正确读取）
Authorization: Bearer eyJhbGci... ← 身份验证令牌（JWT）
Accept: application/json     ← 我希望服务器返回 JSON 格式
                                 ← ③ 空行：必须！分隔头部和体
{                                ← ④ 请求体：实际要提交的数据
  "name": "张三",
  "email": "zhangsan@example.com"
}

/* ====== 第2步：服务器处理后，服务器 → 客户端 (HTTP 响应报文) ====== */

HTTP/1.1 201 Created         ← ① 状态行：HTTP/1.1 + 201(创建成功) + Created
Content-Type: application/json; charset=utf-8  ← ② 响应头开始
Content-Length: 96            ← 响应体字节数
Location: /api/users/42      ← 新创建资源的 URL（201 时常用）
Set-Cookie: session=xyz; HttpOnly; Secure; SameSite=Lax
Cache-Control: no-store      ← 不允许缓存（敏感数据）
                                 ← ③ 空行
{                                ← ④ 响应体：服务器返回的数据
  "id": 42,
  "name": "张三",
  "email": "zhangsan@example.com",
  "createdAt": "2025-07-15T10:30:00Z"
}

HTTP 消息的通用结构（请求和响应都这样组织）


/*
 * HTTP 消息结构（不管是请求还是响应）：
 *
 *  ┌──────────────────────────────┐
 *  │ 起始行 (Start-Line)           │ → 请求 = 方法 + URI + 协议版本
 *  │                               │   响应 = 协议版本 + 状态码 + 原因短语
 *  ├──────────────────────────────┤
 *  │ 头部字段 (Header Fields)      │ → 零个或多个，每个占一行
 *  │ Host: api.example.com         │   格式：名称: 值
 *  │ Content-Type: application/json│
 *  │ ...                           │
 *  ├──────────────────────────────┤
 *  │ 空行 (一个 CRLF，即 \r\n)      │ → 必须有！分隔头部和体
 *  ├──────────────────────────────┤
 *  │ 消息体 (Message Body)         │ → 可选（GET/HEAD 没有）
 *  │ {"name":"张三"}               │   格式由 Content-Type 决定
 *  └──────────────────────────────┘
 *
 * 五个重要规则：
 * 1. HTTP/1.1 请求必须包含 Host 头（因为一台服务器可能托管多个网站）
 * 2. 头部字段名不区分大小写（Content-Type = content-type）
 * 3. 每行结尾必须是 CRLF（\r\n，回车+换行），不能只用 LF（\n）
 * 4. 头部和体之间必须有一个空行
 * 5. HTTP/1.1 默认 Keep-Alive，一个 TCP 连接可以发多个请求/响应
 *
 * 延伸：HTTP/2 和 HTTP/3
 * - HTTP/2：二进制帧，头部压缩，多路复用（一个连接并行多个请求）
 * - HTTP/3：基于 QUIC（UDP 而非 TCP），更快，0-RTT 握手
 */
